## 에이전트 문서 작성법
에이전트 문서는 길게 쓰는 것보다:
- 뭘 해야 하는지
- 뭘 하면 안 되는지
- 언제 끝난 걸로 보는지
를 짧고 분명하게 적는 게 더 중요함.
> 사람도 그렇지만, 에이전트도 기준이 없으면 금방 흐트러짐.
> 굳이 에이전트를 사용하지 않고 웹에서 질문형으로 사용 한다고 해도 이 컨셉은 상당히 유효함.

## `페르소나`
- 말투를 꾸미는 장식이 아님
- 판단 기준과 작업 태도를 고정하는 장치
예를 들면
- 보수적으로 판단하는지
- 바로 구현하는 타입인지
- 먼저 구조를 정리하는 타입인지
이런 걸 정해 줘야 결과가 덜 흔들림.

예시
```md
## Persona
- 빠르게 구현하되, 구조가 무너지면 바로 멈추고 정리한다.
- 파일 책임이 섞이는 상황을 허용하지 않는다.
- 필요 이상으로 추상화하지 않고, 지금 필요한 범위에서만 설계한다.
```

## `네거티브 프롬프트`
- 하지 말아야 하는 행동 목록
- 품질 하한선을 지키는 장치
예를 들면:
- 없는 사실 지어내지 말 것
- 파일 경계를 함부로 넘지 말 것
- 설명만 길게 하고 구현을 미루지 말 것
> 잘하라는 말보다, 뭘 금지할지 적는 쪽이 훨씬 강하게 먹힘.
예시 문구:
```md
## Negative Prompt
- 존재하지 않는 API, 클래스, 기능을 지어내지 말 것.
- 한 파일에 여러 책임을 몰아넣지 말 것.
- 설명만 길게 하고 구현을 미루지 말 것.
- 규칙을 어기는 임시방편 코드를 정답처럼 제시하지 말 것.
```

## `완료 조건`
- 어디까지 하면 끝인지 미리 적는 것
없으면 에이전트가
- 덜 끝내고 멈추거나
- 반대로 쓸데없이 과하게 작업함
그래서 최소한
- 어떤 산출물이 있어야 하는지
- 어떤 검증이 끝나야 하는지
는 적어 두는 게 좋음.
예시 문구:
```md
## Done When
- 요구된 파일이 생성되거나 수정되어 있다.
- 왜 이렇게 바꿨는지 짧게 설명할 수 있다.
- 필요한 검증이나 확인이 끝나 있다.
- 남은 위험이나 미완료 항목이 있으면 분명하게 적혀 있다.
```

## 에이전트의 역할 분리
한 에이전트가:
- 조사하고
- 설계하고
- 구현하고
- 검수까지
전부 혼자 들고 있으면 맛이 잘 감.
가능하면
- 조사하는 에이전트
- 구현하는 에이전트
- 검토하는 에이전트
처럼 역할을 나누는 게 안정적임.

사용 예시:
- Research Agent: 관련 규칙, 참고 자료, 기존 구조만 정리한다.
- Build Agent: 실제 코드 수정과 파일 작성만 담당한다.
- Review Agent: 책임 분리 위반, 누락, 회귀 가능성만 확인한다.

창을 3개 띄워놓고 쓰던, 세션을 3개로 유지하면서 쓰던
혹은 설계 > 리셋 > 구현 > 리셋 > 리뷰 > 리셋 > 설계 > 리셋 > 구현 > 리셋....을 반복하던
아무튼 분리하는게 좋음.

## 주기적인 컨텍스트 리셋이 왜 필요한가
- 에이전트는 지금까지 들어온 입력을 계속 끌고 감
- 앞에서 넣었던 규칙, 예시, 잡담, 시행착오도 뒤에 남음
이걸 여기서는 그냥 암기 테스트처럼 생각하면 편함.
쉽게 말하면:
- 새 입력만 혼자 들어가는 게 아님
- 앞에서 오간 대화도 같이 다시 들어간다고 보면 됨
- 그래서 대화가 길수록, 다음번에 같이 딸려 들어가는 덩어리도 커짐
예를 들어:
- 1번째 입력: `안녕하세요`
- 1번째 응답: `네 안녕하세요 반가워요`
- 2번째 입력: `저녁 메뉴 추천 좀 해주세요`
그러면 2번째 턴쯤에는 대충 이런 식의 묶음이 같이 들어간다고 보면 됨:
```text
안녕하세요
네 안녕하세요 반가워요
저녁 메뉴 추천 좀 해주세요
```
답변이 나오겠지?
```text
안녕하세요
네 안녕하세요 반가워요
저녁 메뉴 추천 좀 해주세요
비빔밥, 돈까스, 칼국수 같은 메뉴가 괜찮습니다
```
여기서 3번째 입력으로:
- `고추장 못 먹어요, 매운 건 빼고요`
를 넣으면, 그때 입력되는 데이터는 대충 이런 식이 됨
```text
안녕하세요
네 안녕하세요 반가워요
저녁 메뉴 추천 좀 해주세요
비빔밥, 돈까스, 칼국수 같은 메뉴가 괜찮습니다
고추장 못 먹어요, 매운 건 빼고요
```
즉:
- 한 번 말할 때마다 이번 입력만 추가되는 게 아니라
- 이전 대화가 어느 정도 같이 딸려 들어오고
- 그래서 전체 컨텍스트 길이가 계속 커짐
이 감각은 거의 등차수열의 합처럼 이해해도 됨.
- 처음엔 짧음
- 다음 턴엔 조금 더 길어짐
- 또 다음 턴엔 더 긴 기록이 같이 들어감
대화가 길어질수록:
- 초반 규칙은 뒤로 밀리고
- 중요하지 않은 잡담도 자리를 차지하고
- 예전 지시를 지금도 계속 붙잡고 있기 쉬움

그래서 중간중간 하던 작업의 주제나 내용이 바뀌면 다시 /clear건 뭐건 해야함.

## `AGENTS.md` 또는 `CLAUDE.md`의 역할
이 문서는 보통:
- 저장소의 공통 규칙
- 이 프로젝트에서 중요한 금지사항
- 파일 경계와 작업 원칙
을 적는 기준 문서임.
세션마다 매번 길게 설명하지 않게 해 주는, 프로젝트용 공통 브리핑 문서라고 보면 됨.
예시 문구:
```md
## Repository Rules
- UI는 값 표시와 입력 전달만 담당한다.
- PlayerController는 조정만 맡고 세부 기능을 직접 소유하지 않는다.
- 기능 추가 전에는 책임을 먼저 나누고 파일 경계를 확인한다.
- 세션마다 이 문서를 기본 규칙으로 따른다.
```

## 작성할 때 주의할 점
- 추상적인 좋은 말만 적지 말 것
- 실제로 금지할 행동을 적을 것
- 파일명, 폴더명, 책임 경계를 구체적으로 적을 것
- 너무 길게 써서 아무도 안 읽는 문서로 만들지 말 것
짧더라도 바로 행동으로 옮길 수 있게 써야 함.
예시 문구
```md
## Writing Notes
- 좋은 코드 작성 같은 추상 문장만 쓰지 말 것.
- 실제 클래스명, 폴더명, 금지 동작을 적을 것.
- 읽는 사람이 바로 행동할 수 있는 수준으로 짧고 구체적으로 쓸 것.
- 규칙이 많아지면 우선순위를 함께 적을 것.
```

## 디렉터리별로 나눌 필요가 있나
필요한 경우가 많음.
- 루트 문서: 프로젝트 전체 공통 규칙
- 하위 디렉터리 문서: 그 폴더에서만 적용되는 추가 규칙
예를 들면:
- `UI/`는 표시와 입력 전달만
- `Player/`는 플레이어 내부 책임만
- `Editor/`는 런타임 코드 침범 금지
이런 식으로 폴더별 규칙을 따로 두면 훨씬 덜 섞임.
> 전체 규칙 하나만으로는, 폴더별 맥락까지 다 잡기 어려움.
> 또한, 지나치게 긴 전체 규칙 하나는, 컨텍스트 낭비가 심해짐
예시 문구:
```md
## Directory Rules
- `/AGENTS.md`: 저장소 전체 공통 규칙
- `/Assets/Scripts/UI/AGENTS.md`: UI는 표시와 입력 전달만 담당
- `/Assets/Scripts/Player/AGENTS.md`: 플레이어 내부 책임과 파일 경계 규칙
- `/Assets/Scripts/Editor/AGENTS.md`: 에디터 코드는 런타임 코드 책임을 침범하지 않음
```
